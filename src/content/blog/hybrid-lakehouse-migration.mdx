# The Hybrid Data Lakehouse: Designing and Migrating Into a Snowflake + Databricks Architecture

## When One Platform Stops Being Enough

For the past decade, most large enterprises answered the "where does our data live" question by picking a single cloud data warehouse and committing. That worked well enough when the dominant workload was SQL analytics for BI dashboards and reports. It has stopped working — not because the warehouse got worse, but because the *mix of workloads* changed.

A modern data organization now has to simultaneously serve several distinct workload categories, each with different tooling requirements:

- **SQL analytics and BI** — consistent, low-latency, concurrency-friendly queries for dashboards and self-service exploration
- **Large-scale data engineering** — heavy transformation jobs over terabytes of raw history, often more cost-effective on distributed compute engines than on a SQL-optimized warehouse
- **Data science and ML** — distributed Python, feature engineering, experiment tracking, model training, and GPU-accelerated notebooks
- **Operational and AI workloads** — feature serving, model inference, embedding pipelines, and vector operations that pull from governed enterprise data

No single engine is best in class at all four. Snowflake's SQL performance, concurrency handling, and governance are hard to beat for BI and analytical workloads. But it is an expensive place to run bulk ETL and not the natural home for notebook-first data science work. Databricks is optimized for Spark-based engineering and ML, with a toolchain (MLflow, Mosaic AI, GPU clusters) built for data scientists — but its SQL and concurrency story, while much improved, still has gaps for traditional BI workloads at enterprise scale.

The reality is that the roles themselves are converging. Data engineers increasingly need notebook environments and distributed Python. Data scientists need governed, production-quality enterprise data rather than shadow extracts. ML engineers need feature pipelines that read from the same domain structures that BI dashboards consume. The tools that serve these converging personas don't all live in one platform — and pretending otherwise leads to compromises that get more expensive every year.

This article is about how to architect and migrate into a world where you run both platforms intentionally, with a shared data foundation underneath. In a [previous post](https://www.rejozmathew.com/blog/enterprise-data-modernization), I walked through the domain-centric, four-layer architecture we built to modernize an enterprise analytics environment — Source Data Warehouse, Domain Integrated Structures, Consumption Preparation, and Semantic Layer. That architecture holds up remarkably well in a hybrid world. It also makes the migration tractable: the layers give you natural seams to cut along, which is exactly what you want when you're transitioning a production data estate onto new engines without a big-bang cutover.

## The Forces Behind the Shift

Before the *how*, the *why*. Several forces are simultaneously pushing enterprises toward dual-platform architectures, and understanding them shapes every downstream design choice.

![Three workload categories — BI and SQL analytics, data engineering and ETL, and ML and AI — mapped to their best-fit engines, with a shared Apache Iceberg foundation on cloud object storage](/images/blog/hybrid-lakehouse-migration/hybrid-rationale.svg)

Figure 1: Different workloads have different best-fit engines. A shared open-format storage layer is what makes running both economical.

**Workloads have diverged beyond what one engine can serve well.** This is the fundamental driver. If every workload in the enterprise were SQL analytics, a single warehouse would suffice indefinitely. But the growth of data engineering pipelines, ML training, and operational AI means the workload mix now spans SQL, distributed Python, notebook-first experimentation, and large-scale batch processing. Forcing all of these through a SQL-optimized engine means paying compute premiums for workloads that don't benefit from the optimization, and constraining teams with tooling that wasn't designed for their work.

**Persona convergence demands tool diversity.** The traditional separation between "the BI team uses the warehouse" and "the data science team uses notebooks" is dissolving. Data engineers write PySpark and need governed catalog access. Data scientists need production-quality domain data, not CSV extracts. ML engineers need feature pipelines that are consistent with the metrics BI teams report on. These converging roles need access to the *same governed data* through *different tools* — which is exactly the problem a shared open-format storage layer solves.

**Open table formats make dual-platform architectures economically viable.** Running two platforms used to mean running two copies of the data: one in the warehouse's proprietary format, one in a lake for engineering and science. Open table formats — particularly Apache Iceberg — eliminated that penalty. A single physical copy of the data lives in cloud object storage and can be read and written by multiple engines. Storage sits at raw cloud-provider rates rather than warehouse-managed premiums. Bulk ETL that used to run on expensive warehouse compute can move to Spark on spot-priced instances. The economics shift from "two platforms = double the cost" to "two platforms, one storage bill, compute optimized per workload."

**Cost is real, but not sufficient alone.** If cost reduction were the only driver, you could convert your warehouse tables to Iceberg format within the same warehouse and capture storage savings without adding a second platform. The dual-platform architecture is justified when the *workload mix and tooling requirements* demand it — cost savings are a welcome side effect of getting the architecture right, not the primary motivation.

## Open Table Formats: Iceberg, Delta, and the Choice That Depends on Your Center of Gravity

It's worth addressing the format question directly, because the industry narrative — "Iceberg won the format wars" — is both directionally correct and misleadingly simple.

What actually happened: Databricks acquired Tabular (the commercial Iceberg company) in 2024 and announced the long-term convergence of Delta Lake and Iceberg. Snowflake contributed Polaris as an open Iceberg catalog. Both vendors now read and write Iceberg natively. The Iceberg v3 specification adopted features (deletion vectors, row lineage) that had previously been Delta-only differentiators. The *interoperability standards* have clearly converged on Iceberg as the common interface.

But that does not mean Delta Lake is dead or wrong. For a Databricks-native shop where most workloads run on Spark, Delta Lake remains the better choice — it's tightly optimized for the Databricks engine, and UniForm can expose Delta tables as Iceberg to external readers without maintaining two copies. The "which format" question is not an industry-wide answer; it depends on which engine is your center of gravity.

For a **Snowflake-first enterprise** adding Databricks for engineering and ML workloads — which is our situation — **Iceberg is the clear choice**. It's what Snowflake reads natively and performantly. It's what the Iceberg REST Catalog API was designed to serve. And it avoids the indirection of writing Delta and converting to Iceberg for every cross-engine read.

## The Open Lakehouse Stack: Storage, Format, Catalog, Engines

With the format question settled for our context, here is what the hybrid architecture actually looks like at the component level.

![The open lakehouse stack — cloud object storage at the base, Apache Iceberg table format above it, a catalog and governance coordination layer, and Snowflake and Databricks as compute engines reading and writing the same tables](/images/blog/hybrid-lakehouse-migration/open-lakehouse-stack.svg)

Figure 2: The four layers of an open lakehouse. Storage and format are open by design; the coordination layer is what makes multi-engine governance work.

**Cloud object storage** is the base. Parquet data files and Iceberg metadata sit in S3, ADLS, or GCS buckets owned by the enterprise. This is the canonical copy of the data. No engine owns it; the enterprise does.

**The open table format** — Apache Iceberg — sits on top of the raw files and provides ACID transactions, schema evolution, partition evolution, time travel, and row-level updates. The Iceberg v3 specification added deletion vectors and row lineage, which brought performance for heavy MERGE operations close to proprietary formats. The spec is engine-neutral; any compliant engine can read or write.

**The catalog and governance coordination layer** is where the architecture gets interesting — and where most hybrid designs either succeed or fail. I'll dedicate the next section to this, because it's the decision that shapes everything else.

**The compute engines** sit at the top. In our architecture, that means Snowflake and Databricks. The choice of engine becomes a *workload-by-workload* decision: Snowflake for SQL analytics, BI serving, and consumption-layer queries; Databricks for bulk ETL, feature engineering, ML training, and notebook-first data science.

## Governance Coordination: The Architecture Decision That Matters Most

If the choice of engine per layer is the visible part of the hybrid design, the governance strategy is the invisible part that determines whether any of it actually works.

The challenge is straightforward to state and hard to solve: Snowflake has Horizon. Databricks has Unity Catalog. Both are mature, capable governance products. Neither is a superset of the other. They have different RBAC models, different policy semantics, different lineage granularity, and different audit trail formats. If you define masking rules in Horizon for Snowflake and separately in Unity Catalog for Databricks, and both are edited independently, you will have divergent security postures on the same physical tables. In a regulated environment, that is not merely inconvenient — it's a compliance risk.

The temptation is to pick one catalog as "authoritative" and federate the other into it. That works in theory but runs into practical limits quickly. If you pick Unity Catalog as primary, you're handing governance authority to the newer, less-established platform in a Snowflake-first organization — and Snowflake's catalog federation treats external Iceberg tables as read-only, which constrains the consumption layer. If you pick Horizon as primary, Databricks' foreign catalog federation is also read-only, which constrains the engineering layer. Symmetric federation sounds good in vendor keynotes; in practice, each platform's federation is oriented to make *itself* the center.

The more durable answer is a **governance coordination layer** — a platform like Collibra, Atlan, or a purpose-built internal system — that sits *above* both catalogs and serves as the policy source of truth.

This coordination layer does not replace either catalog. Both Unity Catalog and Horizon continue to operate natively within their respective engines. What the coordination layer does is:

- **Define governance policies once** — masking rules, access controls, data classification, retention policies — in a vendor-neutral model
- **Push those policies down** to each engine's native catalog, translated into its native constructs (Horizon tags and row access policies for Snowflake, Unity Catalog ACLs for Databricks)
- **Stitch lineage end-to-end** — a Spark job writes an Iceberg table, Snowflake reads it via catalog integration and materializes a Dynamic Table, and the coordination layer maintains the full provenance chain across both engines
- **Detect and alert on drift** — if a policy is changed directly in one catalog without flowing through the coordination layer, surface it as an anomaly

This architecture also ages well. If a third engine enters the picture later — a streaming engine, a specialized query tool — the coordination layer absorbs it without restructuring the governance model. And in regulated industries, having a vendor-neutral governance record that doesn't live inside either compute vendor's product is a defensible position with auditors and examiners.

## Rethinking the Four-Layer Architecture for a Hybrid World

This is where the layered architecture from the [previous article](https://www.rejozmathew.com/blog/enterprise-data-modernization) pays off. Because each layer has a distinct purpose and a distinct rate of change, each one can be placed on the engine that best fits its workload — without rebuilding the whole stack.

![The four-layer architecture — Source Data Warehouse, Domain Integrated Structures, Consumption Preparation, and Semantic Layer — mapped onto a hybrid Snowflake and Databricks environment, with Iceberg as the shared storage format across layers 1 through 3](/images/blog/hybrid-lakehouse-migration/four-layer-hybrid.svg)

Figure 3: The four layers map cleanly onto a hybrid environment. Each layer keeps its purpose; only the engine and storage choice change.

### Layer 1 — Source Data Warehouse on Iceberg

The Source Data Warehouse is a natural fit for Iceberg on object storage, managed primarily by Databricks. It's the largest layer by volume, the most write-heavy (every source system change lands here), and the workload is almost pure data engineering: ingest, date-chain, assign surrogate keys, store. Spark on Databricks is cost-effective for exactly this profile — large batch writes, occasional full-table rescans, and schema evolution as new source systems come online. Writing these tables as Iceberg means Snowflake can read them for the occasional analytical need via catalog integration, but the expensive bulk load work happens on the more cost-effective engine.

One subtlety: the SDW's historical tracking (date-chaining and surrogate key generation) benefits from Iceberg v3's row-level update capabilities. What used to require rewriting entire partitions when a late-arriving update came in can now be expressed as a MERGE with deletion vectors, making the ingestion layer more efficient at scale.

### Layer 2 — Domain Integrated Structures on Iceberg

Domain Integrated Structures are the transformation product — cleansed, conformed, business-friendly entity and event data. This is the layer that gets the most varied downstream consumption: BI tools pulling point-in-time snapshots, data scientists training models, and ML pipelines constructing feature sets. That variety is the tell. *Anything that multiple workload types will consume should sit in the open format*.

In practice, this means building the domain transformations in Databricks (where the Spark-based metadata-driven pipeline framework lives), writing the outputs as Iceberg tables, and exposing them to Snowflake via catalog integration. Snowflake's role at this layer is as a reader, not a writer — and that asymmetry is fine, because it matches the natural data flow direction.

The rate-of-change lesson from the previous article still applies here, and it matters more in a hybrid world: domain structures, when built properly with elemental and comprehensive attributes, change least frequently. That stability is what makes them a safe place to draw the storage-format boundary. If you pick a volatile, narrow, report-driven structure to put in Iceberg, you'll feel every schema change. Well-designed domain structures absorb change at the edges while the core remains stable.

### Layer 3 — Consumption Preparation: Where the Hybrid Gets Real

The Consumption Preparation layer is where the architecture gets most nuanced. This layer's job is to produce time-windowed, aggregated, analysis-ready structures — the kind of views that dashboards and downstream analytics hit constantly with tight latency requirements.

Here, a practical pattern emerges: Snowflake's Dynamic Tables reading from Iceberg source tables. The domain-integrated source lives on object storage in Iceberg; Snowflake's Dynamic Table materializes a hot cache with a declared refresh lag for fast, concurrent BI serving.

An honest acknowledgment: **this means the architecture is not literally "one copy of the data."** The canonical copy is in Iceberg. The Dynamic Tables are derived caches optimized for a specific consumption pattern. That's a meaningful architectural improvement over the old world — where you had full independent copies with independent ETL and no shared lineage — but it's not zero duplication. The right framing is *one canonical source of truth with purpose-built serving caches*, and the governance coordination layer's job is to ensure lineage and provenance flow from the canonical copy through every cache.

One discipline to enforce: don't cache everything. A Dynamic Table is only cost-effective when the query-to-materialized-row ratio is high — roughly ten or more queries per materialized refresh cycle as a rule of thumb. If a cache is refreshed constantly but queried rarely, you're burning warehouse compute for nothing. The cost efficiency of this layer depends on being selective about what gets cached versus what gets read directly from Iceberg.

For data science and ML workloads, the Iceberg source remains the primary access path — they typically don't need the cached view, they need the raw elemental data at full grain for feature engineering and model training.

### Layer 4 — The Semantic Layer, Engine-Agnostic

The Semantic Layer is the one place where the hybrid architecture simplifies rather than complicates. A well-built semantic layer is already abstracted from its underlying physical storage — users interact with *product*, *customer*, *revenue* as concepts, and the semantic tool resolves those to queries against whichever engine holds the relevant data. In the hybrid world, the semantic layer routes queries to the appropriate engine based on where the data sits, transparently.

The Feature Store pattern extends the same idea to ML consumption. Features derived from domain-integrated Iceberg tables can be served to training jobs on Databricks and online inference systems without a second physical copy — the feature values are computed once from the canonical domain structures and consumed everywhere.

## The Migration Path: Strangler, Not Big Bang

The single most important decision in a migration like this is not technical — it's sequencing. Migrations that try to cut over everything in one event almost always fail; the organization cannot absorb that much simultaneous change, and the rollback surface area is impossibly large. The layered architecture gives you a natural way to avoid that trap.

![Three-phase migration timeline showing the progression from a fully Snowflake-native environment, through a transition phase where the Source Data Warehouse and Domain Integrated Structures have moved to Iceberg while Snowflake retains consumption and semantic workloads, to a target hybrid state with workload-specialized engines reading and writing a shared Iceberg foundation](/images/blog/hybrid-lakehouse-migration/migration-path.svg)

Figure 4: A strangler-pattern migration. Move the heaviest and most stable layers first; let the consumption layers follow when the economics justify it.

**Phase 1 — Before.** The starting state: everything lives in a single warehouse, in the warehouse's proprietary format. All four layers run on the same engine and share the same bill. The warehouse cost is growing, and the engineering and data science teams are either paying for expensive compute they don't love, or silently building shadow environments on the side.

**Phase 2 — Transition.** The approach that has worked for us is to migrate bottom-up: **start with the Source Data Warehouse, then move the Domain Integrated Structures, and leave the Consumption Preparation and Semantic Layers in place until last**. Three reasons.

First, the SDW is the largest layer by volume and the heaviest on bulk-write compute. Moving it to Iceberg on Databricks delivers a meaningful and fast cost improvement, which helps fund the rest of the program. Second, the SDW is the furthest upstream of business users — its migration is nearly invisible to the business, which is exactly the kind of change you want to do first. Third, once the SDW is in Iceberg, the Domain Integrated Structures become easier to migrate because their inputs are already in the target format.

During Phase 2, the Snowflake-facing consumption and semantic layers continue to run largely unchanged, but now reading from Iceberg sources via catalog integration rather than from Snowflake-native tables. End users see no change. Dashboards don't break. BI tools keep querying the same semantic objects. Behind the scenes, the storage-and-engine substrate has shifted; the business-facing contract has not.

The bottom-up approach also naturally limits the governance complexity. The SDW and Domain Integrated Structures are *producers* — they write data, they don't consume from other areas. That means the catalog coordination problem at the bottom two layers is simpler: Databricks writes Iceberg, Snowflake reads it. You don't need bidirectional catalog federation for these layers, which buys time to mature the governance coordination layer before the more complex consumption-tier decisions arrive.

**Phase 3 — Target.** With the SDW and Domain Integrated Structures established on Iceberg, the Consumption Preparation layer gets its own review. Some structures stay on Snowflake as Dynamic Tables (high-concurrency BI, tight latency, complex SQL). Others migrate to Databricks or remain as Iceberg-native structures (ML feature prep, engineering transformations). The Semantic Layer, if it was built to be engine-agnostic, requires minimal change — its job is to resolve business terms to physical queries, and in the hybrid world it just routes to whichever engine holds the target structure.

A critical discipline throughout: **keep the quality and observability infrastructure running continuously across the transition**. The metadata-driven validation, applicability conditions, and centralized message log described in the previous article do not require rewriting for a hybrid environment — they run wherever the pipeline runs. If anything, the hybrid architecture makes observability *more* important, because lineage now crosses engine boundaries. Every transformation, every catalog integration read, every Dynamic Table refresh needs to carry provenance forward — and this is where the governance coordination layer earns its keep, stitching lineage across engines into a single auditable chain.

## Risks and Anti-Patterns

A few things will bite you if you don't plan for them.

**Performance parity is a myth, at least for now.** Iceberg reads are meaningfully slower than native-format reads on most engines. Even Snowflake has a roughly 20% performance penalty on Iceberg versus native tables. This is a manageable cost, but not a free one. Workloads with strict query-latency SLAs will still want a materialized cache in the warehouse-native format, which is exactly why the Dynamic Table pattern in Layer 3 exists. Know where the penalty matters and cache accordingly.

**Governance drift is the real risk, not catalog choice.** The danger isn't picking the wrong catalog — it's having policies defined independently in both catalogs that gradually diverge. A masking rule added in Horizon but not propagated to Unity Catalog means the same table has different security postures depending on which engine queries it. The governance coordination layer exists specifically to prevent this. Treat any policy change that doesn't flow through the coordination layer as an incident, not a convenience.

**Don't federate what you could materialize.** Query federation — joining tables across engines at runtime — is attractive for ad-hoc exploration and terrible for production workloads. It pushes compute across engine boundaries, and the network and egress costs can dwarf the savings the hybrid architecture was supposed to deliver. The discipline is: materialize the join into Iceberg and let each engine read the result, rather than paying two engines to do the join every time.

**Egress is the silent killer.** Moving data between engines in different cloud regions will generate egress charges that add up fast. Plan the storage regionality carefully, keep the Iceberg data and the consuming engines co-located, and design for egress minimization from the start.

**Don't migrate what isn't worth migrating.** A surprising amount of data in most enterprise warehouses is either unused, stale, or available upstream. A migration is a forcing function for a data-estate cleanup. The tables nobody queries can be decommissioned. The legacy ETL outputs that duplicate upstream sources can be deleted. Every row that doesn't move is a row you don't have to validate in the target architecture. I have never seen a migration regret its lightening-up choices; I have often seen programs regret their scope creep.

## What We Learned

**Layered architectures age well.** The four-layer, domain-centric framework was not designed with hybrid lakehouses in mind, and yet it maps cleanly onto this paradigm. The separation of concerns it enforces — raw history versus curated domains versus consumption-ready versus business semantics — turns out to be exactly the separation you need when you want to place each layer on the engine that fits its workload. Good architecture earns compound returns over time, even when the underlying technology shifts.

**Govern above the engines, not inside them.** Trying to make one vendor's catalog the universal governance authority for a multi-engine architecture is a losing proposition. Each catalog is optimized for its own engine and makes assumptions the other engine doesn't share. A coordination layer above both — maintaining policy as the source of truth and pushing it down to each engine's native catalog — is more work to set up and dramatically more durable.

**Migrate bottom-up, preserve the business contract.** End users should not feel the storage or engine transition. The Source Data Warehouse moves first because it's invisible to them; the Semantic Layer moves last — or barely at all — because any change there is visible everywhere. In between, the Consumption Preparation layer is where the most interesting hybrid decisions happen and where the cost-latency trade-offs live.

**"One copy" means one canonical source, not zero caches.** The value of the open-format architecture is a single governed source of truth with traceable lineage — not the elimination of all derived structures. Dynamic Tables, feature serving layers, and BI caches will exist. What changes is that they all derive from the same canonical Iceberg source, through a lineage chain the governance coordination layer can audit. That's the real improvement over the old world, where independent copies had independent ETL and no shared provenance.

**Observability must cross engine boundaries.** In a single-engine world, lineage is a feature of the warehouse. In a hybrid world, lineage and quality logging have to be engine-agnostic and catalog-aware. Every transformation should write to a lineage record that a downstream consumer can walk back to a validated source regardless of which engine produced the output along the way. If you treat observability as platform-specific, you lose the audit chain precisely when regulators and auditors care about it most.

**Pick the format that matches your center of gravity.** For a Snowflake-first enterprise adding Databricks capabilities, Iceberg is the clear choice — it's what Snowflake reads natively, and it avoids the indirection of Delta-to-Iceberg conversion. For a Databricks-first organization, Delta Lake with UniForm remains a perfectly valid approach. The format choice is a function of your architecture, not an industry-wide proclamation.

**Don't mistake format openness for vendor independence.** Committing to Iceberg reduces storage-format lock-in. It does not, by itself, reduce catalog, tooling, or operational lock-in. Each of those requires its own deliberate strategy — and the governance coordination layer is how you keep that optionality real rather than theoretical.

## Conclusion

The hybrid data lakehouse isn't a destination so much as a re-stabilization. For the last decade, the right answer was "pick one cloud data warehouse and go deep." For the next stretch, it's "pick the right engine for each workload, keep the data in an open format underneath, and coordinate governance above both." That's a more demanding architecture to design and operate, but it produces a more resilient and more capable one — and it fits the shape of modern engineering, science, and analytics work in a way that a single-engine environment cannot.

If you've already built a layered, domain-centric data architecture, you have far more of the work done than you probably realize. The layers are the migration seams. Each one moves on its own schedule, to its own best-fit engine, and the business-facing contract stays intact throughout. The hardest part of the transition isn't technical — it's sequencing, governance, and the discipline to migrate bottom-up without being tempted by the visible wins at the top of the stack.

The platform choices will keep shifting. The layered architecture underneath them, governed by a coordination layer above them, is what keeps those shifts from being catastrophic. That's the real investment — the one that lets you treat Snowflake, Databricks, and whatever comes next as tools you choose rather than platforms you're stuck with.
